## 1. Основания для разработки

### 1.1. Документы-основания
Разработка системы выполняется в рамках учебного проекта по дисциплине "Безопасность информационных систем" для обеспечения практического применения знаний в области криптографии и защиты информации.

Настоящее техническое задание разработано в соответствии с требованиями ГОСТ 34.602-2019 "Информационные технологии. Комплекс стандартов на автоматизированные системы. Техническое задание на создание автоматизированной системы".

### 1.2. Наименование темы разработки
Система для подписи и проверки целостности файлов с использованием асимметричной криптографии.

## 2. Назначение и цели создания системы

### 2.1. Назначение системы
Система предназначена для обеспечения целостности файлов посредством создания цифровых подписей и их последующей верификации. Система позволяет гарантировать, что файл не был изменен с момента подписания.

### 2.2. Цели создания системы
- Создание инструмента командной строки для работы с цифровыми подписями
- Реализация поддержки современных алгоритмов асимметричного шифрования (RSA, ECDSA)
- Обеспечение простоты использования и интеграции в существующие процессы
- Демонстрация применения криптографических методов на практике

## 3. Характеристики объектов автоматизации

### 3.1. Краткие сведения об объектах автоматизации
Объектами автоматизации являются файлы произвольного типа и размера, для которых требуется обеспечить целостность. Система должна работать с файлами любых размеров, используя потоковую обработку.

### 3.2. Сведения об условиях эксплуатации
- **Режим работы**: интерактивный (командная строка)
- **Тип пользователя**: администраторы, разработчики, конечные пользователи
- **Среда выполнения**: Windows, Linux, macOS

## 4. Требования к системе

### 4.1. Требования к системе в целом

#### 4.1.1. Требования к структуре системы
Система должна быть реализована как консольное приложение с модульной архитектурой:
- Отдельный модуль для генерации ключей
- Отдельный модуль для криптографических операций
- CLI интерфейс для взаимодействия с пользователем

#### 4.1.2. Требования к функциям системы
Система должна обеспечивать выполнение следующих функций:
1. **Генерация ключевых пар**
   - Поддержка алгоритма RSA с настраиваемым размером ключа
   - Поддержка алгоритма ECDSA с выбором эллиптической кривой
   - Сохранение ключей в формате PEM

2. **Создание цифровой подписи**
   - Вычисление хеш-суммы SHA-256 файла
   - Подпись хеш-суммы закрытым ключом
   - Сохранение подписи в текстовом формате (Base64)

3. **Проверка цифровой подписи**
   - Вычисление хеш-суммы проверяемого файла
   - Загрузка и декодирование подписи
   - Верификация подписи открытым ключом
   - Вывод результата проверки

#### 4.1.3. Требования к видам обеспечения
- **Программное обеспечение**: Go 1.25.4 или выше
- **Аппаратное обеспечение**: стандартные требования для работы Go приложений
- **Информационное обеспечение**: файлы ключей в формате PEM, файлы подписей в формате Base64

### 4.2. Требования к функциям системы

#### 4.2.1. Функция генерации ключей
**Входные данные**:
- Алгоритм (RSA или ECDSA)
- Для RSA: размер ключа в битах (2048, 3072, 4096)
- Для ECDSA: название кривой (p256, p384, p521)
- Пути для сохранения ключей

**Выходные данные**:
- Файл закрытого ключа в формате PEM
- Файл открытого ключа в формате PEM

**Обработка**:
- Генерация криптографически стойкой пары ключей
- Кодирование ключей в формат PEM
- Сохранение ключей в указанные файлы
- Установка прав доступа 0600 для закрытого ключа

#### 4.2.2. Функция создания подписи
**Входные данные**:
- Путь к файлу закрытого ключа
- Путь к подписываемому файлу
- Путь для сохранения подписи

**Выходные данные**:
- Файл с Base64-кодированной подписью

**Обработка**:
- Потоковое чтение файла и вычисление SHA-256
- Загрузка закрытого ключа из PEM файла
- Подпись хеш-суммы соответствующим алгоритмом
- Кодирование подписи в Base64
- Сохранение подписи в файл

#### 4.2.3. Функция проверки подписи
**Входные данные**:
- Путь к файлу открытого ключа
- Путь к проверяемому файлу
- Путь к файлу подписи

**Выходные данные**:
- Сообщение о результате верификации (успех/неудача)

**Обработка**:
- Потоковое чтение файла и вычисление SHA-256
- Загрузка открытого ключа из PEM файла
- Декодирование подписи из Base64
- Верификация подписи соответствующим алгоритмом
- Вывод результата

### 4.3. Требования к видам обеспечения

#### 4.3.1. Требования к математическому обеспечению
- Использование стандартных криптографических алгоритмов
- Применение криптографически стойких генераторов случайных чисел
- Корректная реализация алгоритмов подписи и верификации

#### 4.3.2. Требования к информационному обеспечению
- Формат ключей: PEM (совместим с OpenSSL)
- Формат подписей: Base64 (текстовый)
- Кодировка файлов: UTF-8

#### 4.3.3. Требования к лингвистическому обеспечению
- Интерфейс командной строки: английский язык
- Сообщения об ошибках: английский язык
- Документация: русский язык

#### 4.3.4. Требования к программному обеспечению
- Язык программирования: Go 1.25.4
- Использование только стандартной библиотеки Go
- Кроссплатформенность (Windows, Linux, macOS)

### 4.4. Требования к информационной безопасности

#### 4.4.1. Требования к защите информации
- Закрытые ключи должны храниться с минимальными правами доступа
- Использование криптографически стойких алгоритмов
- Защита от утечки закрытых ключей через файловую систему

#### 4.4.2. Требования к надежности
- Обработка всех возможных ошибок
- Валидация входных данных
- Защита от переполнения буферов
- Корректная обработка больших файлов (потоковая обработка)
